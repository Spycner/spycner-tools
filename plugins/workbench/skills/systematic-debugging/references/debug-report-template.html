BUG-2026-0142

Stale cache served to logged-in users after deploy

SEVERITY: HIGH Resolved Reported 2026-05-06 Resolved 2026-05-08 Owner on-call

Summary

Bug, impact, resolution

After the v3.18 release, roughly 8 percent of authenticated users saw stale dashboard data for up to 30 minutes. The CDN was serving a public response variant to logged-in clients because the Vary header dropped Cookie on a refactored response builder.

Impact: customer-reported confusion, three support tickets, no data corruption. Resolution: restored Vary: Cookie on the dashboard handler and added a contract test that asserts the header on every authenticated route.

Hypothesis tree

Initial signal: dashboard widget showed yesterday's totals for a subset of users. Hypotheses explored, in order, with results:

  • refuted Database replication lag is serving old reads Replica lag metrics flat at < 80 ms across the window. Direct DB queries returned the correct, fresh values.
    • refuted A specific replica is slow Per-replica lag dashboards normal. Affected users were not pinned to a single replica.
  • confirmed CDN is serving a cached public response to logged-in users Reproduced by curling the dashboard with and without the auth cookie, observing identical bodies and an age header up to 1800.
    • confirmed The response is missing Vary: Cookie Headers from curl -I show only Vary: Accept-Encoding. Pre-release responses included Vary: Accept-Encoding, Cookie.
    • refuted CDN edge config dropped the Vary on Cookie rule Edge config diff was empty across the deploy window. Header was missing at the origin, not stripped at the edge.
  • pending Other authenticated routes may be affected too Tracked as a followup audit. See Followups.

Root cause

The v3.18 refactor introduced a shared build_response() helper. Its vary parameter accepted a single string and defaulted to "Accept-Encoding". The dashboard handler was migrated to this helper but was not updated to pass the additional Cookie token, so the response shipped with Vary: Accept-Encoding only. The CDN, doing exactly what that header asks for, treated public and authenticated responses as a single cache key.

app/http/responses.py
  1. def build_response(body, status=200, vary="Accept-Encoding"):
  2. headers = {
  3. "Content-Type": "application/json",
  4. "Vary": vary,
  5. }
  6. return Response(body, status=status, headers=headers)
  8. def dashboard(request):
  9. payload = render_dashboard(request.user)
  10. return build_response(payload)
See fix below, app/http/responses.py:9

Fix

Two-part change. First, pass the correct vary value at the dashboard handler. Second, add a contract test so any future authenticated handler that drops Cookie from Vary fails CI.

app/http/responses.py (diff)
  1. def dashboard(request):
  2. payload = render_dashboard(request.user)
  3. - return build_response(payload)
  4. + return build_response(payload, vary="Accept-Encoding, Cookie")
tests/contract/test_vary_header.py (new)
  1. +def test_authenticated_routes_vary_on_cookie(client, auth_user):
  2. + for route in AUTHENTICATED_ROUTES:
  3. + resp = client.get(route, user=auth_user)
  4. + vary = resp.headers.get("Vary", "")
  5. + assert "Cookie" in vary, f"{route} missing Cookie in Vary: {vary!r}"
