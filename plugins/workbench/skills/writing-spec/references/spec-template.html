Problem

[Describe the user pain or system gap in concrete, observable terms.] Today, the workflow forces contributors to keep context in their head across multiple tools, which causes handoffs to drop information and reviewers to repeat questions that earlier discussion already answered.

[Cite the evidence that this is worth solving now: support tickets, recent incidents, qualitative feedback, or measured friction.] Without intervention, the cost compounds as the team grows and the surface area expands.

Solution overview

[Two or three sentences naming the chosen approach and the shape of the change.] We introduce a single canonical artifact that captures the decisions made during brainstorming and exposes them to downstream tooling through stable anchors.

The artifact is a self-contained HTML document. It renders in any browser without a build step, ships next to the skill that produces it, and remains readable as plain source. Downstream consumers depend only on the section anchors listed in the table of contents.

User stories

The following stories describe whole-task outcomes. Each one is independently shippable and testable. [Tighten or split stories until each fits in a single review.]

  1. As a contributor, I open the spec and immediately see status and authorship.

    The header surfaces title, date, status, and author near the top of the viewport so a reviewer landing cold can orient in under five seconds. [Acceptance: header visible above the fold at 1280 by 800.]

  2. As a reviewer on a phone, I can jump between sections without losing my place.

    On narrow viewports the table of contents collapses to the top of the page; on wider viewports it sticks to the side. [Acceptance: TOC anchors all resolve, no horizontal scroll at 320px.]

  3. As a downstream tool, I can extract sections by stable id.

    Every required section uses a documented id. A consumer that needs only the risks block can fetch the document and select #risks without parsing prose. [Acceptance: ids match the schema check in CI.]

Testing decisions

We verify the template through filesystem checks rather than rendered output. The checks run fast and stay deterministic, and they catch the regressions that downstream consumers actually care about.

  • The file is non-empty and starts with a valid HTML5 doctype or <html> tag.
  • The source is free of em-dash (U+2014) and en-dash (U+2013) characters.
  • Every required section id is present so downstream extractors do not break.

Out of scope

We name the boundaries of this spec so reviewers do not propose work that belongs in a follow-up.

  • JavaScript-driven interactivity such as live editing or collapsible TOC behaviour beyond CSS.
  • Server-side rendering or templating engines. The artifact is hand-edited HTML.
  • Localization. The template ships in English. [Track translation as a separate spec when needed.]

Risks

Each risk lists its trigger, its impact, and the mitigation we accept up front. Unlisted risks default to the team's standard escalation path.

  1. The template drifts from the section schema downstream tools expect.

    If a section id is renamed without coordination, downstream extractors silently drop content. We mitigate this with a CI check that asserts the full id list. [Owner: the team that ships the extractor.]

  2. Authors paste em-dash characters from word processors.

    Pasted em-dashes break the project's writing-style lint. The same CI check that gates the section ids also greps for U+2014 and U+2013, so a regression fails the build rather than reaching review.

  3. Dark mode rendering reads poorly on some monitors.

    The palette uses muted greens and warm neutrals tuned for both schemes, but contrast remains environment-dependent. [Mitigation: spot-check on at least one OLED panel and one matte IPS panel before release.]
